Module 01 — Physics
Stability, Forces and Motion
01
The Inverted Pendulum
The physical model behind every self-balancing robot
PhysicsStabilityCenter of MassTorque
▾

A self-balancing robot is physically identical to an inverted pendulum on a cart. Unlike a regular pendulum that is stable at the bottom, an inverted pendulum is inherently unstable at the top. Gravity constantly tries to topple it, and the robot must apply continuous corrections to stay upright.

Figure 1.1 — Inverted Pendulum Model
Core Concepts
01
Center of Mass (CoM): Must remain directly above the wheel contact point. Any deviation creates a torque demand on the motors proportional to the displacement.
02
Gravitational Torque: When tilted by angle θ, gravity produces τ = m·g·L·sin(θ). The motors must produce an equal and opposite torque at all times.
03
Dynamic Equilibrium: The robot is never truly stationary. It makes constant micro-corrections — a higher loop frequency means smaller corrections per cycle and smoother balance.
04
Small Angle Approximation: For θ below about 15°, sin(θ) ≈ θ in radians. This linearises the physics, making PID control effective near the balance point.
τ = m · g · L · sin(θ)
τapprox = m · g · L · θ    (for small θ)
m = mass (kg)  ·  g = 9.81 m/s²  ·  L = height to CoM (m)  ·  θ = tilt angle (rad)

Module 02 — Sensors
Measuring Tilt — IMU and Sensor Fusion
02
IMU and Sensor Fusion
Combining accelerometer and gyroscope for a stable angle estimate
IMUSensor FusionComplementary FilterDrift
▾

The IMU combines two sensors with complementary weaknesses. The accelerometer is accurate when still but noisy during motion. The gyroscope is smooth and fast but drifts over time. Fusion gives a stable, accurate angle in real-time.

Figure 2.1 — Sensor Fusion Block Diagram
Accelerometer
Measures linear acceleration. Tilt from gravity: θ = atan2(ax, az). Accurate when stationary. Weakness: motor vibration produces large noise spikes — raw data unusable during motion.
Gyroscope
Measures angular velocity (deg/s). Integrating gives angle. Responds instantly. Weakness: integration drift — tiny errors accumulate into degree-level offset within seconds.
θfused = α·(θprev + ω·Δt) + (1−α)·θaccel
α ≈ 0.98 — 98% gyroscope (speed) + 2% accelerometer (drift correction)

Code — Complementary Filter (C / Arduino)
C / Arduino
// Run in main loop at fixed dt (5 ms = 200 Hz)
float alpha = 0.98f, dt = 0.005f, angle = 0.0f;

void update_angle(float gyro_dps, float ax, float az) {
  float gyro_a  = angle + gyro_dps * dt;
  float accel_a = atan2f(ax, az) * (180.0f / M_PI);
  angle = alpha * gyro_a + (1.0f - alpha) * accel_a;
}

Module 03 — Control Theory
PID Controller
03
PID Control Theory
Proportional, Integral and Derivative — gains, tuning and anti-windup
PIDClosed LoopGain TuningAnti-Windup
▾

PID calculates motor correction from the error between the desired angle (0°, upright) and the actual angle. It runs at 100–200 Hz. Getting the three gains wrong produces oscillation, steady lean, or noise.

Figure 3.1 — PID Closed-Loop Control Block Diagram
P — Proportional
Output = Kp × error. Reacts to current tilt. Too high causes rapid oscillation; too low and the robot falls slowly.
I — Integral
Output = Ki × accumulated error. Corrects a steady lean such as an off-centre battery. Too high causes slow hunting oscillation.
D — Derivative
Output = Kd × rate-of-change of error. Damps overshoot. Too high amplifies sensor noise and causes twitching.
u(t) = Kp·e(t) + Ki·∫e dt + Kd·(de/dt)
u(t) → motor PWM value  ·  e(t) = setpoint − measured angle

Code — PID with Anti-Windup (Python)
Python
class PID:
    def __init__(self, kp, ki, kd, dt, lim=255):
        self.kp,self.ki,self.kd,self.dt,self.lim=kp,ki,kd,dt,lim
        self.integral=self.prev_error=0.0

    def compute(self, setpoint, measured):
        e=setpoint-measured
        P=self.kp*e
        self.integral=max(-100,min(100,self.integral+e*self.dt))
        I=self.ki*self.integral
        D=self.kd*(e-self.prev_error)/self.dt
        self.prev_error=e
        return max(-self.lim,min(self.lim,P+I+D))

pid=PID(kp=25.0,ki=0.8,kd=4.5,dt=0.005)
print(pid.compute(0.0,3.5))  # about -87.5

Module 04 — Actuators
Motors, PWM and Drivers
04
PWM and Motor Drivers
Translating PID output into physical wheel movement
PWMDC MotorL298NDuty Cycle
▾

MCU GPIO pins output only 3.3–5 V at ~40 mA — too weak to drive a motor. A motor driver (L298N) amplifies this. Speed is controlled via PWM: rapidly switching ON and OFF; the duty cycle (ON ratio) sets average power to the motor.

Figure 4.1 — PWM Duty Cycles
DC motors are the standard choice for self-balancing bots. The L298N driver accepts two direction pins (IN1, IN2) and one PWM speed pin (ENA), enabling full forward, reverse and brake control from the microcontroller.

Code — Motor Driver (Arduino)
C / Arduino
const int ENA=9, IN1=7, IN2=8;

void motor(int pwm) {
  int spd=constrain(abs(pwm),0,255);
  if(pwm>0){digitalWrite(IN1,HIGH);digitalWrite(IN2,LOW);}
  else if(pwm<0){digitalWrite(IN1,LOW);digitalWrite(IN2,HIGH);}
  else{digitalWrite(IN1,LOW);digitalWrite(IN2,LOW);}
  analogWrite(ENA,spd);
}
void loop(){motor((int)pid.compute(0.0,getAngle()));delay(5);}
